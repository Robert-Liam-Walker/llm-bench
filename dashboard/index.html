llm-bench

Per-task accuracy and cost across three Claude tiers (Opus 4.7, Sonnet 4.6, Haiku 4.5) and three open-weight models served by Groq (Llama 3.3 70B, DeepSeek R1 distill, Llama 3.1 8B). Half the tasks are deterministically scored (unit tests, SQL execution, regex match, exact answer); the rest use a disclosed LLM-as-judge rubric (Opus grades, 0–1 scale). Re-run on the 1st of every month so drift across model updates is visible.

Methodology, briefly

  • code_gen · model writes a Python function; runner exec's it and runs unit tests · deterministic
  • code_debug · diagnose + fix buggy code · judged on rubric
  • code_refactor · improve working code without changing behavior · judged on rubric
  • sql · query against a SQLite fixture; result rows compared · deterministic
  • regex · pattern applied to held-out match/non-match examples · deterministic
  • json_extract · jsonschema validation + key field checks · deterministic
  • math · word problem; final-number exact match (1‰ tolerance) · deterministic
  • science · multi-step factual reasoning · judged
  • summarize · short/medium/long article → summary · judged
  • email · workplace email from scenario · judged on tone + completeness
  • logic · puzzle with one correct answer · deterministic
  • edge_cases · should hedge / refuse / identify trick · judged

Caveat the judge: when Opus grades Opus's own outputs, that row is self-judging on the six judge-scored tasks. The deterministic tasks (the other six) are unaffected, and the Groq models are judged by an independent family.
